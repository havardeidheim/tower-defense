Codebase Review: Refactoring Opportunities
1. GameScene is a God Class (578 lines)
GameScene.ts handles input, state management, rendering, UI coordination, sound effects, and game logic all in one class. This is the single biggest maintainability issue — any change requires understanding the whole file.

Suggestion: Extract concerns into focused helpers:

GameSceneInput — mouse/keyboard handlers
GameSceneRenderer — all render methods
Keep GameScene as orchestrator
2. No Single Source of Truth for Entity Stats
Tower stats are defined in three places:

Hardcoded in tower subclass constructors/applyUpgrade() methods
TowerStatsConfig.ts (UI display only — decorative, not used by game logic)
GameScene.ts:523-527 (range preview values duplicated again)
Enemy stats are similarly hardcoded in each subclass's initStats(). Some share identical formulas (NormalEnemy/FastEnemy both use 100 + 100 * level, DodgeEnemy/SuperEnemy both use 70 + 70 * level).

Suggestion: Create a single config-driven stats system. Define all base stats, upgrade paths, and costs in one place. Have entity classes read from config instead of hardcoding.

3. ~50+ Magic Numbers Scattered Everywhere
Examples:

UI layout positions throughout UIManager.ts and WavePreview.ts
Health bar dimensions (barWidth - 10, barHeight = 4, barY = y - 8)
Suggestion: Extract to constants.ts or create focused config files (UILayout.ts, ColorScheme.ts). This is the single most impactful change for AI development — named constants make intent clear.

7. Structural Types Instead of Interfaces
Both UIManager.ts and InfoDisplay.ts use anonymous structural types for tower info:

tower: { getType: () => string; damage: number; range: number; ... }
Suggestion: Define an ITowerInfo interface and use it consistently. This makes the contract explicit and helps AI understand the expected shape.

8. Tower Instantiation Switch Statement
GameScene.ts:338-344 has a switch on tower type strings. If TowerType is extended, the compiler won't catch the missing case.

Suggestion: Create a TowerFactory with a Record<TowerType, (x, y) => Tower> map. Exhaustive by construction.


Priority Ranking for AI-Friendliness
Priority	Refactoring	Impact
1	Extract magic numbers to named constants	Every file benefits; AI can understand intent
2	Consolidate entity stats into config	Single source of truth; fewer files to touch
3	Split GameScene into focused classes	Reduces context needed for any change
4	Create TowerFactory + ITowerInfo interface	Type safety catches errors at compile time

The codebase is well-structured at the directory level and uses good patterns (template method, abstract base classes, scene management). The main issues are about consolidation — pulling scattered magic values, duplicated logic, and implicit contracts into explicit, centralized, well-typed constructs. These changes would dramatically reduce the context an AI needs to safely make modifications.
