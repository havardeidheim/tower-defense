Codebase Review: Refactoring Opportunities
Updated: 2026-03-07

============================================================
PRIORITY 1 — Quick Wins (minimal effort, clear benefit)
============================================================

A. Extract ITowerInfo Interface
   Status: DONE
   Files: UIManager.ts:214, InfoDisplay.ts:47
   Both use an anonymous structural type for tower info:
     tower: { getType: () => string; damage: number; range: number; attackSpeed: number; level: number; maxLevel: number }
   Fix: Define ITowerInfo interface once, use it in both places.
   Effort: Tiny. Impact: Clearer contracts for AI.

B. Fix AreaTower Hardcoded Slow Duration
   Status: DONE
   File: AreaTower.ts:37
   Passes literal 1600 instead of SLOW_DURATION constant from constants.ts.
   Fix: Import and use SLOW_DURATION.
   Effort: One-line change.

C. Deduplicate Tower Upgrade Costs
   Status: DONE
   Files: NormalTower.ts:16, AreaTower.ts:18, SpreadTower.ts:18, PoisonTower.ts:19
   All four towers have identical getUpgradeCost(): costs = [40, 80, 120].
   Fix: Move to Tower base class as default implementation.
   Effort: Small.

============================================================
PRIORITY 2 — Medium Impact (moderate effort)
============================================================

E. Eliminate Tower Range Preview Duplication
   Status: NOT DONE
   File: GameScene.ts:530-533
   Tower ranges hardcoded in preview: { 'Normal': 140, 'Area': 80, 'Spread': 140, 'Poison': 120 }
   These duplicate the getBaseRange() values in each tower subclass.
   Fix: Either instantiate a temp tower to query range, or add static ranges
   to a config. Best approach: add base ranges to TOWER_STATS in TowerStatsConfig.ts
   and read from there.
   Effort: Small. Impact: Eliminates silent desync risk.

F. Deduplicate ShieldEnemy/SuperEnemy Shield Logic
   Status: NOT DONE
   Files: ShieldEnemy.ts:22-32, SuperEnemy.ts:24-34
   Identical takeDamage() and takePoisonDamage() methods in both classes.
   Fix: Extract to a shared helper function or mixin:
     function applyShieldedDamage(health: number, amount: number, reduction: number): number
   Both classes call the helper. Keeps subclass structure intact.
   Effort: Small. Impact: One place to change shield behavior.

G. Tower/Enemy Factory Maps
   Status: NOT DONE
   Files: GameScene.ts:344-349 (tower switch), WaveManager.ts:112-129 (enemy switch)
   Both use switch statements for entity instantiation.
   Fix: Replace with Record<TowerType, (x, y) => Tower> and
   Record<string, (x, y, level) => Enemy> maps. Exhaustive by construction.
   Effort: Small. Impact: Compiler catches missing cases when types are added.

============================================================
PRIORITY 3 — Larger Refactors (higher effort, strategic)
============================================================

H. Consolidate Entity Stats into Config
   Status: NOT DONE
   Problem: Tower stats live in 3 places:
     1. Tower subclass methods (getBaseDamage, applyUpgrade) — actual game logic
     2. TowerStatsConfig.ts — UI display only (decorative)
     3. GameScene.ts:530-533 — range preview (duplicated)
   Enemy stats similarly hardcoded per-subclass initStats().
   Some share formulas: Normal/Fast both use 100+100*level, Dodge/Super use 70+70*level.
   Fix: Create a single source of truth. Tower subclasses read base stats and
   upgrade formulas from config. TowerStatsConfig and preview both derive from it.
   Effort: Medium. Impact: Highest — single file to edit for balance changes.

I. GameScene Decomposition
   Status: PARTIALLY DONE (UIManager, InfoDisplay, WavePreview already extracted)
   File: GameScene.ts (584 lines)
   What remains is orchestration: game state, entity lifecycle, input routing, rendering.
   Each method is focused and the file is readable.
   Verdict: Lower priority than originally assessed. Further splitting (GameSceneInput,
   GameSceneRenderer) would add indirection without proportional benefit. The file is
   manageable as-is. Revisit if it grows past ~700 lines.
